# MICo-Bench Evaluation

MICo-Bench is the evaluation benchmark for Multi-Image Composition (MICo). It contains **897 cases** across four tasks:

| Task | JSONL | Cases | Description |
|---|---|---|---|
| Object-Centric | `mico_bench_object_centric.jsonl` | 138 | Objects В± scenes |
| Human-Centric | `mico_bench_human_centric.jsonl` | 168 | Persons В± scenes |
| HOI | `mico_bench_hoi.jsonl` | 291 | Person + objects/clothes |
| De&Re | `mico_bench_dere.jsonl` | 300 | Decompose & recompose |

## Metric: Weighted-Ref-VIEScore

The final score for each case is computed as:

\[
\text{Score} = W \times \text{SC} \times \text{PQ}
\]

Where:

- **W (Weight)** вЂ” The fraction of source images correctly preserved in the generated output. Faces are verified via ArcFace cosine similarity; non-face elements (objects, clothes, scenes) are verified via a VLM (e.g. Qwen2.5-VL-72B).

- **SC (Semantic Consistency)** вЂ” GPT-5.4 compares the generated image against a human-verified reference image (not against all source images, to avoid cross-image attention overload). It outputs two sub-scores:
  - *Prompt Following (PF)*: how well the generated image follows the text prompt.
  - *Subject Resemblance (SR)*: how well it matches the reference image.
  - SC = в€љ(PF Г— SR)

- **PQ (Perceptual Quality)** вЂ” GPT-5.4 evaluates the generated image alone for:
  - *Naturalness*: coherent lighting, shadows, geometry, realism.
  - *Artifacts*: absence of distortions, duplicated limbs, unnatural textures.
  - PQ = в€љ(Naturalness Г— Artifacts)

## JSONL Format

Each case in the JSONL files contains:

```json
{
  "idx": 0,
  "input": ["mico_bench_hoi/0/0.jpg", "mico_bench_hoi/0/1.png"],
  "reference": "mico_bench_hoi/0/reference.jpg",
  "Task": "HOI",
  "editing_type": "1 Human + 1 Object",
  "prompt": "A gentleman sits comfortably on a cushioned seat ...",
  "human_indices": [0]
}
```

- `input`: Paths to source images (relative to `MICo-Bench/`).
- `reference`: Path to the human-verified reference image.
- `human_indices`: Which positions in `input` are human portraits (used for ArcFace-based weight computation).
- De&Re cases use `ed_idx` instead of `idx`, and additionally include `refined_prompt`.

## Step-by-Step Evaluation

### Step 0: Generate Images

Run your model on every case in all four JSONL files. For each case, save the generated image as:

```
results/{model_name}/{task_name}/{case_id}.png
```

- `{task_name}`: one of `mico_bench_object_centric`, `mico_bench_human_centric`, `mico_bench_hoi`, `mico_bench_dere`
- `{case_id}`: the `idx` field (or `ed_idx` for De&Re)
- Format: `.png`, `.jpg`, or `.webp`

### Step 1: Compute Weights

For each case, compute the weight W вЂ” the fraction of source images correctly preserved in the generated output.

The verification method depends on the source type (determined by `human_indices`):

**Human faces** (positions listed in `human_indices`):

Extract face embeddings from both the source portrait and the generated image using [ArcFace](https://github.com/deepinsight/insightface). Compute cosine similarity between the best-matched face pair. A face is considered preserved if the similarity exceeds the threshold:
- **0.50** for Person-Centric tasks
- **0.45** for all other tasks (HOI, De&Re)

This yields a continuous similarity score per face, but for weight computation it is binarized: 1 if above threshold, 0 otherwise.

**Non-face inputs** (objects, clothing, scenes вЂ” all positions NOT in `human_indices`):

Use a VLM (e.g. Qwen2.5-VL-72B) to perform a binary check. Feed the source image and the generated image together, and ask:

```
Given two images:
- Image 1 is a source image containing a specific element.
- Image 2 is an AI-generated composite image.

Does the element from Image 1 appear in Image 2?
Answer only "yes" or "no".
```

Each "yes" counts as 1, each "no" counts as 0.

**Computing W:**

```
W = (number of preserved sources) / (total number of source images)
```

For example, if a case has 4 source images (1 face + 3 objects), the face passes the ArcFace threshold, 2 objects are confirmed by the VLM, and 1 object is not found, then W = 3/4 = 0.75.

**Save format:**

Save weights as a JSON file per task under `results/{model_name}/weights/`:

```
results/{model_name}/weights/{task_name}.json
```

```json
{
  "0": 0.75,
  "1": 1.0,
  "2": 0.5
}
```

Each key is the case ID (string), each value is W в€€ [0, 1].

### Step 2: Compute SC, PQ, and Final Score

Run the evaluation script:

```bash
python eval_score.py \
    --model_name  your_model \
    --task        all \
    --api_key     sk-xxx \
    --base_url    https://api.openai.com/v1
```

Arguments:

| Argument | Default | Description |
|---|---|---|
| `--model_name` | (required) | Model name, matching the directory under `results/` |
| `--task` | `all` | Specific task or `all` |
| `--api_key` | (required) | OpenAI API key |
| `--base_url` | `https://api.openai.com/v1` | API base URL (supports OpenAI-compatible endpoints) |
| `--gpt_model` | `gpt-5.4` | GPT model to use as evaluator |
| `--results_dir` | `MICo-Bench/results/` | Root results directory |

The script will:
1. Read each case from the JSONL.
2. Load the pre-computed weight (defaults to 1.0 if no weight file is found).
3. Call GPT-5.4 to compute PQ (generated image only).
4. Call GPT-5.4 to compute SC (reference + generated image).
5. Compute the final score: W Г— SC Г— PQ.
6. Save per-case scores to `results/{model_name}/scores/{task_name}.json`.
7. Save the aggregated summary to `results/{model_name}/summary.json`.

The script supports **incremental evaluation** вЂ” if interrupted, re-running it will skip already-scored cases.
